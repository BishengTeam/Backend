# 小程序办学许可证的解决方案

> 客户：四川智天远科技有限公司
> 业务：售卖考试券（职业认证报名）+ 网课（在线视频课程）
> 现状：甲方无民办学校办学许可证，无法申请微信小程序教育/培训类目
> 编制日期：2026-05-16

---

## 一、方案概述

甲方无办学许可证但要开展考试报名+网课业务，微信小程序端面临类目准入障碍。以下提供三条可选路径：

| 方案 | 核心思路 | 周期 | 成本 | 法律确定性 |
|------|---------|------|------|-----------|
| 方案A：申请办学许可证 | 补齐资质，正规入驻教育培训类目 | 3-6 个月 | 中等（行政费用+时间成本） | 最高，一劳永逸 |
| 方案B：与有资质机构合作 | 甲方做信息服务+报名支付，合作方承担教学交付 | 2-4 周 | 中等（分润/服务费） | 高，需合作协议+业务隔离 |
| 方案C：三端分离绕审 | 教育信息服务类目 + 课程外置H5 + 审核期隐藏 | 1-2 周 | 低（纯技术实现） | **低，存在法律合规风险（见第七章）** |

## 二、方案A：申请办学许可证

### 2.1 适用场景

- 业务规模较大，长期经营预期明确
- 甲方有固定经营场所和教学管理人员
- 愿意投入行政合规成本

### 2.2 申请流程

```
准备材料 → 向教育主管部门提交申请 → 现场核查 → 审批发证 → 微信小程序类目变更
```

### 2.3 关键前置条件

| 条件 | 要求 | 备注 |
|------|------|------|
| 办学场地 | 符合消防安全、面积要求 | 各地标准不一，咨询当地教育局 |
| 教学管理人员 | 校长任职资格、教师资格证 | 需提前储备或聘任 |
| 注册资金 | 各地标准 10-100 万不等 | 货币出资，实缴 |
| 办学章程 | 规范的学校章程 | 可参照模板 |
| 消防验收 | 办学场地消防验收合格 | 住建部门出具 |

### 2.4 获证后的技术落地

办学许可证到手后,可直接申请微信小程序「教育服务 → 教育培训」类目，无需再绕审：

- 小程序内直接承载课程内容（视频/直播/题库）—— 无需 web-view 外跳
- 考试券以「认证考试报名费」直接售卖 —— 教育培训类目下的正常业务
- 无需 `audit_mode` 开关 —— 审核期与正式期内容一致
- 可在微信小店开设教育培训店铺 —— 扩展销售渠道

### 2.5 后端技术架构

办学许可证到位后的简化架构：

```
┌─ 小程序端（类目：教育培训）──────┐
│  ├── 认证列表 / 报名下单 / 支付     │  ← 全套功能，无需隐藏
│  ├── 在线课程（视频/直播/题库）      │  ← 直接承载教学内容
│  └── 个人中心 / 订单 / 学习记录      │  ← 全功能
└──────────────────────────────────┘
              │
              ▼
┌─ 后端 API ────────────────────────┐
│  ├── 用户 / 课程 / 订单 / 支付     │
│  ├── 学习进度 / 题库 / 考试         │
│  └── 客服 / 退款 / 发票            │
└──────────────────────────────────┘
```

> 获证后架构大幅简化，无需外链 H5、`audit_mode` 开关、URL 签名等绕审组件。

## 三、方案B：与有资质培训机构合作

### 3.1 核心原则

**业务分离 + 服务外包**：甲方小程序内只做信息展示和报名收费，教学交付由合作方（持有办学许可证的培训机构）完成。双方之间的法律关系和业务边界通过合作协议明确。

### 3.2 角色分工

```
┌── 甲方（四川智天远）────────────┐
│  角色：认证信息服务商              │
│  职责：                           │
│  ├── 小程序运营（教育信息服务类目） │
│  ├── 认证类型介绍 / 考试资讯       │
│  ├── 报名费代收（自建收银台）       │
│  ├── 用户引流 + 客服前置           │
│  └── 向合作方采购教学服务           │
└──────────────────────────────────┘
              │
              │ 教学服务采购协议
              ▼
┌── 合作方（持证培训机构）─────────┐
│  角色：教学服务提供方              │
│  职责：                           │
│  ├── 课程内容制作与更新            │
│  ├── 线上教学交付（含直播/录播）    │
│  ├── 题库维护 + 模考系统            │
│  ├── 学员答疑 + 学习督导            │
│  └── 颁发培训结业证明               │
└──────────────────────────────────┘
```

### 3.3 技术架构

```
┌─ 小程序端（类目：教育信息服务）───┐
│  ├── 认证列表 / 考试介绍           │  ← 纯信息展示，类目合规
│  ├── 报名下单 + 微信支付            │  ← 付费产品名=「认证报名服务费」
│  ├── 课程入口 → web-view 跳转 H5   │  ← 教学内容承载在合作方域名
│  └── 个人中心 / 订单 / 联系客服     │
└──────────────────────────────────┘
         │                    │
         ▼                    ▼
┌─ 甲方后端 API ─┐   ┌─ 合作方教学 H5 ──────────┐
│  订单/支付/      │   │  承载全部课程内容          │
│  用户/报名       │   │  合作方独立部署/独立域名    │
│  课程访问鉴权    │   │  甲方通过 SSO token 跳转    │
└────────────────┘   └──────────────────────────┘
```

### 3.4 课程交付链路的合规实现

与绕审方案的关键区别：**不隐藏服务内容，不欺骗平台审核**。

```
用户报名付费（小程序内微信支付，商品名=「认证报名服务费」）
    │
    ▼
甲方后端生成课程访问凭证（HMAC-SHA256 + 过期时间戳）
    │
    ▼
甲方调用合作方 SSO 接口，用户自动登录合作方教学平台
    │
    ▼
用户在合作方平台学习课程（URL 为合作方备案域名）
    │
    ▼
学习进度回传甲方（用于订单履约确认）
```


### 3.6 合作协议关键条款

与合作方的协议至少需覆盖以下内容：

| 条款 | 内容要点 |
|------|---------|
| 合作范围 | 甲方负责招生引流，合作方负责教学交付 |
| 资质保证 | 合作方保证办学许可证有效，并承担资质不合规产生的损失 |
| 结算方式 | 按学员人数/课程套餐分润，或固定服务费 |
| 品牌露出 | 教学平台明确标注合作方品牌，小程序端标注教学内容提供方 |
| 数据归属 | 学员数据归属约定，隐私合规 |
| 退出机制 | 合作终止后学员的课程交付不受影响（履约保障） |
| 违约责任 | 教学质量、数据安全、资质失效等场景的赔偿条款 |

### 3.7 合作方的选择标准

- 持有有效的《中华人民共和国民办学校办学许可证》
- 办学内容涵盖「职业资格培训」「信息技术培训」等相关类目
- 拥有在线教学平台（含直播、录播、题库功能）
- 具备 SSO 或 API 对接能力
- 有商誉，无行政处罚记录（可通过信用中国查询）

## 四、方案C：三端分离绕审方案


### 4.1 核心策略：三端分离

```
┌─ 小程序端（类目：教育信息服务）────┐
│  不需要办学许可证的类目               │
│  ├── 认证列表 / 考试介绍 / 学信网引导  │  ← 纯信息展示，类目合规
│  ├── 报名下单 + 微信支付（自建收银）   │  ← 支付走小程序内，不依赖小店类目
│  └── 课程入口                         │  ← 仅为入口按钮，不直接承载教学内容
└──────────────────────────────────────┘
         │                    │
         ▼                    ▼
┌─ 后端 API ─────┐   ┌─ 外部 H5 站点 ──────────┐
│  订单/支付/     │   │  承载全部课程内容          │
│  用户/认证      │   │  web-view 加载，审核时不可见 │
│  external_url   │   │  独立部署，独立审核          │
└────────────────┘   └──────────────────────────┘
```

### 4.2 小程序类目选择

#### 目标类目：教育服务 → 教育信息服务

参考案例：H3C 官方小程序使用此分类，成功通过审核并长期运营。

| 对比项 | 教育培训类目 | 教育信息服务类目 |
|--------|------------|----------------|
| 办学许可证 | 需要 | **不需要** |
| 教师资质 | 需要 | **不需要** |
| 视听节目许可 | 需要（五选一） | **不需要** |
| 基础资质 | 执照 + ICP 备案 | 执照 + ICP 备案 |
| 审核难度 | 高（教育主管部门前置审批） | 低（普通企业类目） |
| 适用主体 | 培训学校 / 教育机构 | 普通企业（含非教育行业） |

#### 教育信息服务类目可承载的功能

- 认证类型介绍（H3C NE、深信服 CSE、NISP 等）
- 考试大纲 / 报名须知 / 费用说明
- 学信网验证引导
- 报名信息填写 + 费用支付（自建收银台）
- 课程入口（通过 web-view 跳转外部 H5）

> **关键点**：类目审核关注的是小程序内「展示什么内容」。只要小程序内的页面不直接承载教学视频/直播，以信息展示和报名服务为主，即可通过教育信息服务类目审核。

### 4.3 考试券售卖的包装方式

微信平台（微信小店侧）禁止以「考试券」「代金券」名义直接售卖。但在小程序自建交易场景下，可通过以下包装方式实现：

| 包装方式 | 前端展示文案 | 后端实体 |
|---------|------------|---------|
| 认证报名服务费 | 「H3C NE 认证考试报名」 | `order` 表 + `cert_type` 字段 |
| 考前辅导服务包 | 「深信服 CSE 考前冲刺包」 | `order` + 附赠课程访问权限 |
| 认证一条龙服务 | 「NISP 一级认证全流程服务」 | `order` + 报名代办 + 课程学习 |

#### 定价策略建议

| 套餐 | 包含内容 | 定位 |
|------|---------|------|
| 基础报名 | 考试费代缴 + 考位查询 | 引流款 |
| 标准备考 | 报名 + 课程 + 题库 | 主推款 |
| VIP 全流程 | 报名 + 课程 + 题库 + 1v1 督导 + 不过免费重学 | 利润款 |

> 套餐设计的本质：**考试报名费是刚需入口，课程和附加服务是利润来源**，但在类目审核层面只暴露「认证信息服务」。

### 4.4 课程交付的绕审链路

#### 技术流程

```
用户报名付费（小程序内微信支付）
    │
    ▼
后端生成带签名的 external_url（HMAC-SHA256 + 过期时间戳）
    │
    ▼
前端 web-view 加载 H5 页面（携带 token 参数）
    │
    ▼
H5 页面向后端校验 token → 有效则放行，无效则拒绝访问
```



### 4.6 方案C风险矩阵

| 风险 | 等级 | 触发条件 | 应对措施 |
|------|------|---------|---------|

## 六、建议路径

1. **短期（1-2 周内）**：启动方案B，签约一家持证培训机构，以「教育信息服务」类目上线小程序，课程承载在合作方域名。商品名称为「认证报名服务费」，课程页面标注教学提供方。

2. **中期并行（同时进行）**：启动方案A，准备办学许可证申请材料。方案B 的运营数据（招生规模、收入流水）可作为办学许可证申请的「业务需求证明」。

3. **长期（6-12 个月后）**：办学许可证获批后，申请小程序类目变更为「教育培训」，课程迁移回小程序内，逐步减少对合作方的依赖。

4. **方案C 定位**：仅作为方案B 受阻时的紧急止损选择（如短期内无法找到合适合作方、业务上线时间有硬性 deadline），且需在方案B 就位后立刻迁移，缩短风险敞口。

## 七、法律风险提示（方案C）

方案C 存在以下法律合规风险，决策前需充分知悉：

### 7.1 类目选择的合规性：名实不符

方案C 选择「教育信息服务」类目（无需办学许可证），实际开展「教育培训+考试券售卖」业务。微信小程序的类目规则只是平台自治规则，不替代行政许可。如果实际业务性质是「教育培训」而非「信息服务」，《民办教育促进法》要求的办学许可证义务**不会因为在微信后台勾选了另一个类目而消失**。平台审核侧可能「绕过去」了，但监管侧的法律责任仍在。

### 7.2 「包装」考试券的法律定性

将「考试券」包装为「认证报名服务费」「考前辅导服务包」存在三个层面的法律问题：

**合同层面——虚假陈述**：向消费者展示的商品/服务描述与实际交付内容不一致，可能构成《消费者权益保护法》第五十五条中的消费欺诈——退一赔三。

**行政监管层面——超出核准经营范围**：如果营业执照上写的是「信息技术服务」或「教育信息服务」，实际从事考试代理报名+课程培训，属于超出核准登记的经营范围从事经营活动，面临市场监管部门的行政处罚。

**税务层面**：不同业务类型适用不同税率和发票类目。「教育信息服务」和「教育培训服务」在增值税处理上可能适用不同政策。以错误类目开票存在税务合规风险。

### 7.3 `audit_mode` 开关的法律性质

`audit_mode = true` 时隐藏课程内容，这不是「合规包装」，这是**故意隐瞒真实业务内容，欺骗平台审核机制**。

**对微信平台的违约责任**：违反《微信小程序平台运营规范》，面临小程序下架、主体封禁、结算资金冻结等处罚。

**主观故意的直接证据**：一旦发生消费者投诉或监管调查，`audit_mode` 开关的存在本身就是主观故意的直接证据——证明了明知业务不合规，主动设计了隐瞒机制。

**如果涉及金额较大且持续经营**：隐藏真实服务内容、制造审核通过后切换正式版本的行为模式，已超出「技术规避」范畴，可能被认定为以欺骗手段获取平台服务资源。

### 7.4 风险矩阵评估修正

方案C 原风险矩阵存在低估：

| 风险项 | 原评估 | 修正评估 | 理由 |
|--------|--------|---------|------|
| 审核被识别 | 中 | **高** | 主观故意+系统设计=恶意违规，一次抽检即可触发连锁后果 |
| web-view 举报 | 低 | **中** | 一旦举报涉及消费者权益纠纷，民事+行政双线展开 |
| 类目政策变更 | 低 | 低 | 微信有权单方修改规则，无抗辩空间（维持原评估） |

## 八、方案B 法律注意事项

### 8.1 广告法合规

- 小程序内的「认证报名服务费」描述应与实际交付内容一致
- 不得使用「包过」「保过」「100% 通过」等绝对化用语
- 合作方的教学资质信息需在课程页面真实标注

### 8.2 消费者权益保护

- 报名页面明确公示：报名服务由甲方提供，教学服务由合作方 XXX 培训机构提供
- 退款规则明确区分：报名服务费退款 vs 课程学费退款
- 客服入口清晰，投诉受理路径可追溯

### 8.3 数据合规

- 与合作方的数据共享需在隐私政策中披露
- 学员个人信息传输需做必要脱敏
- 与合作方签署数据处理协议（DPA）
