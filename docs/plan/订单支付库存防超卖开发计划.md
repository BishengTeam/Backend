# 订单支付库存防超卖开发计划

> 编制日期：2026-05-18  
> 适用范围：订单创建、微信支付、库存/名额锁定、支付回调、超时释放、对账补偿

---

## 1. 目标

建立订单支付链路的库存一致性保护，避免以下问题：

- 超卖：多个用户并发下单时，可售库存或报名名额被卖成负数。
- 少卖：未支付订单长期占用库存，导致真实可售名额无法释放。
- 错账：支付成功、退款、重复回调、回调丢失导致本地订单状态与微信支付状态不一致。

核心原则：

```text
下单时锁定库存
支付成功时确认成交
超时/取消时释放库存
回调和对账必须幂等
```

---

## 2. 当前基础

当前仓库已有以下基础能力：

- 订单状态：`pending` / `paid` / `completed` / `refunded` / `closed`。
- 订单状态迁移函数：`apply_order_status_transition()`。
- 商户订单号 `out_trade_no` 唯一。
- 支付回调验签。
- 支付回调按订单行 `with_for_update()` 加锁。
- 支付金额校验：回调金额必须等于订单金额。
- 订单表已预留 `inventory_id`、`expires_at`、`closed_at`、`close_reason` 字段，用于后续库存锁定和超时关闭链路。

当前缺口：

- 没有库存/名额表。
- 下单时未锁定库存。
- 没有超时释放库存任务。
- 没有微信查单对账补偿任务。
- 没有并发库存扣减的 PostgreSQL 集成测试。

---

## 3. 目标状态流

建议将订单状态扩展为：

```text
pending    已创建，库存已锁定，待支付
paid       支付成功，库存已确认为成交
completed 业务完成
closed     超时或取消关闭，库存已释放
refunded   已退款
```

状态迁移：

```text
pending -> paid
pending -> closed
paid -> completed
paid -> refunded
completed -> refunded
closed -> 终态
refunded -> 终态
```

库存流转：

```text
下单 pending:
available - 1
locked + 1

支付成功 paid:
locked - 1
sold + 1

超时关闭 closed:
locked - 1
available + 1

退款 refunded:
第一期默认不恢复 available，仅记录退款库存策略；后续由配置决定是否恢复
```

---

## 3.1 已确认业务规则

第一期库存归属模型采用认证报名名额库存：

- 库存粒度：`cert_type` 级别，一个认证类型对应一份可售报名名额。
- 库存类型：`inventory_type = 'certification'`。
- 关联字段：`ref_code` 存储认证类型代码，与订单请求和订单表中的 `cert_type` 对应，例如 `H3C-NE`。
- 创建订单时，按 `inventory_type = 'certification' AND ref_code = :cert_type` 定位库存对象。
- 第一阶段不按课程批次或考试场次拆分库存；后续可通过新增 `inventory_type = 'course_batch'`、`inventory_type = 'exam_session'` 等类型扩展，不改变库存锁定/确认/释放主流程。

退款库存策略：

- 第一期退款默认不恢复库存，即订单从 `paid` 或 `completed` 进入 `refunded` 后，不执行 `sold - 1`、`available + 1`。
- 退款时应记录库存流水 `refund_no_restore`，用于审计该笔退款未回补可售名额。
- 后续如业务需要，可增加配置决定是否恢复库存；启用恢复时再使用 `refund_restore` 流水动作。

## 4. 数据库设计

### 4.1 库存/名额表

建议新增 `inventory` 表。第一期固定用于认证报名名额库存，按 `cert_type` 级别归属；后续可扩展课程批次或考试场次库存。

字段建议：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | int | 主键 |
| inventory_type | string | 库存类型；第一期固定为 `certification`，后续可扩展 `course_batch` / `exam_session` |
| ref_code | string | 业务对象编码；第一期存储认证类型代码，与 `order.cert_type` 对应 |
| total_quota | int | 总名额 |
| available_quota | int | 可售名额 |
| locked_quota | int | 已锁未支付名额 |
| sold_quota | int | 已售名额 |
| is_active | bool | 是否启用 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

约束建议：

- 第一期开启库存的记录满足 `(inventory_type, ref_code)` 唯一。
- `total_quota >= 0`
- `available_quota >= 0`
- `locked_quota >= 0`
- `sold_quota >= 0`
- `available_quota + locked_quota + sold_quota = total_quota`

### 4.2 库存流水表

建议新增 `inventory_record` 表，用于审计每一次库存变化。

字段建议：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | int | 主键 |
| inventory_id | int | 库存 ID |
| order_id | int | 订单 ID |
| action | string | lock/confirm/release/refund_restore/refund_no_restore |
| quantity | int | 变化数量 |
| before_available | int | 变化前可售 |
| before_locked | int | 变化前锁定 |
| before_sold | int | 变化前已售 |
| after_available | int | 变化后可售 |
| after_locked | int | 变化后锁定 |
| after_sold | int | 变化后已售 |
| reason | string | 变化原因 |
| created_at | datetime | 创建时间 |

### 4.3 订单表扩展

建议扩展 `order` 表：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| inventory_id | int | 对应库存/名额 |
| expires_at | datetime | 支付过期时间 |
| closed_at | datetime/null | 关闭时间 |
| close_reason | string/null | 关闭原因 |

建议新增约束：

- `ck_order_status` 增加 `closed`。
- `transaction_id` 增加唯一索引，防止同一微信交易绑定多个订单。

## 5. 接口和服务改造

### 5.1 创建订单

接口：`POST /api/orders`

改造目标：

- 校验证书类型、实名状态、价格配置。
- 按 `inventory_type = 'certification' AND ref_code = cert_type` 定位认证报名库存对象。
- 在同一个事务中原子锁定库存。
- 创建 `pending` 订单。
- 写入 `expires_at`。
- 写库存流水 `lock`。

原子锁定方式：

```sql
UPDATE inventory
SET available_quota = available_quota - 1,
    locked_quota = locked_quota + 1
WHERE inventory_type = 'certification'
  AND ref_code = :cert_type
  AND available_quota >= 1;
```

如果影响行数为 `0`，返回业务错误：认证报名名额不足。

### 5.2 微信支付统一下单

接口：`POST /api/payment/prepay`

改造目标：

- 只允许当前用户自己的订单发起支付。
- 只允许 `pending` 状态。
- 如果订单已过期，先关闭订单并释放库存，再拒绝支付。
- 微信支付金额继续使用订单表 `price`，不能由前端传入。

### 5.3 支付回调

接口：`POST /api/payment/callback`

保留现有能力：

- 验签。
- 按 `out_trade_no` 查询订单。
- 使用 `with_for_update()` 锁定订单行。
- 校验支付金额。
- 重复回调幂等。

新增能力：

- `pending -> paid` 时，将库存从 `locked` 转为 `sold`。
- 写库存流水 `confirm`。
- 保存 `transaction_id` 和 `paid_at`。
- 如果订单已是 `paid` 或 `completed`，不重复确认库存。
- 如果订单已 `closed` 但微信显示支付成功，进入异常补偿流程，不能静默忽略。

### 5.4 超时关闭任务

新增脚本或定时任务：

```text
扫描 pending 且 expires_at < now() 的订单
锁定订单行
pending -> closed
locked - 1
available + 1
写库存流水 release
```

建议第一期先提供可手动执行脚本，后续接入定时任务。

### 5.5 微信查单对账

新增对账任务：

- 查询长时间未完成的 `pending` 订单。
- 调用微信查单接口确认真实支付状态。
- 微信成功：本地补偿为 `paid`，库存 `locked -> sold`。
- 微信未支付且订单过期：本地关闭并释放库存。
- 微信状态未知：保留 pending，等待下一轮对账或人工处理。

## 6. 并发和幂等要求

必须满足：

- 下单锁库存和创建订单在同一个数据库事务中完成。
- 库存扣减必须用条件更新，不允许先查后扣。
- 支付回调必须先验签再改状态。
- 支付回调必须锁订单行。
- 同一个 `out_trade_no` 重复回调只能推进一次状态。
- 同一个 `transaction_id` 不能绑定多个订单。
- 超时任务和支付回调并发时，最终只能有一个成功推进状态。

## 7. 测试计划

### 7.1 本地无数据库测试

覆盖：

- 订单状态枚举包含 `closed`。
- 合法状态迁移通过。
- 非法状态迁移抛 `ConflictException`。
- `refunded` 和 `closed` 是终态。
- 过期订单不能发起支付。
- 支付回调必须调用验签、行锁、状态迁移。

命令：

```powershell
python -m pytest tests/unit -v
```

### 7.2 PostgreSQL 集成测试

覆盖：

- 并发下单同一库存，不会超卖。
- 库存不足时订单不创建。
- 下单锁库存后订单创建失败时事务回滚。
- 支付成功回调重复触发，只确认一次成交。
- 超时关闭释放库存。
- 支付回调与超时关闭并发时，最终状态只能是 `paid` 或 `closed`。
- `ck_order_status`、`out_trade_no` 唯一、`transaction_id` 唯一约束生效。

命令：

```powershell
$env:TEST_DATABASE_URL="postgresql+asyncpg://<user>:<password>@localhost:5432/<test_db>"
$env:TEST_DATABASE_URL_SYNC="postgresql://<user>:<password>@localhost:5432/<test_db>"
alembic upgrade head
python -m pytest tests/integration/db -v
```
