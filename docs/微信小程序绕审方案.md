# 微信小程序绕审方案

> 客户：四川智天远科技有限公司
> 业务：售卖考试券（职业认证报名）+ 网课（在线视频课程）
> 现状：甲方无民办学校办学许可证，无法申请微信小程序教育/培训类目
> 编制日期：2026-05-16

---

## 一、核心策略：三端分离

```
┌─ 小程序端（类目：教育信息服务）────┐
│  不需要办学许可证的类目               │
│  ├── 认证列表 / 考试介绍 / 学信网引导  │  ← 纯信息展示，类目合规
│  ├── 报名下单 + 微信支付（自建收银）   │  ← 支付走小程序内，不依赖小店类目
│  └── 课程入口                         │  ← 仅为入口按钮，不直接承载教学内容
└──────────────────────────────────────┘
         │                    │
         ▼                    ▼
┌─ 后端 API ─────┐   ┌─ 外部 H5 站点 ──────────┐
│  订单/支付/     │   │  承载全部课程内容          │
│  用户/认证      │   │  web-view 加载，审核时不可见 │
│  external_url   │   │  独立部署，独立审核          │
└────────────────┘   └──────────────────────────┘
```

---

## 二、小程序类目选择

### 目标类目：教育服务 → 教育信息服务

参考案例：H3C 官方小程序使用此分类，成功通过审核并长期运营。

| 对比项 | 教育培训类目 | 教育信息服务类目 |
|--------|------------|----------------|
| 办学许可证 | 需要 | **不需要** |
| 教师资质 | 需要 | **不需要** |
| 视听节目许可 | 需要（五选一） | **不需要** |
| 基础资质 | 执照 + ICP 备案 | 执照 + ICP 备案 |
| 审核难度 | 高（教育主管部门前置审批） | 低（普通企业类目） |
| 适用主体 | 培训学校 / 教育机构 | 普通企业（含非教育行业） |

### 教育信息服务类目可承载的功能

- 认证类型介绍（H3C NE、深信服 CSE、NISP 等）
- 考试大纲 / 报名须知 / 费用说明
- 学信网验证引导
- 报名信息填写 + 费用支付（自建收银台）
- 课程入口（通过 web-view 跳转外部 H5）

> **关键点**：类目审核关注的是小程序内「展示什么内容」。只要小程序内的页面不直接承载教学视频/直播，以信息展示和报名服务为主，即可通过教育信息服务类目审核。

## 三、考试券售卖的合规包装

微信平台（微信小店侧）禁止以「考试券」「代金券」名义直接售卖。但在小程序自建交易场景下，可通过以下包装方式合法实现：

| 包装方式 | 前端展示文案 | 后端实体 |
|---------|------------|---------|
| 认证报名服务费 | 「H3C NE 认证考试报名」 | `order` 表 + `cert_type` 字段 |
| 考前辅导服务包 | 「深信服 CSE 考前冲刺包」 | `order` + 附赠课程访问权限 |
| 认证一条龙服务 | 「NISP 一级认证全流程服务」 | `order` + 报名代办 + 课程学习 |

### 定价策略建议

| 套餐 | 包含内容 | 定位 |
|------|---------|------|
| 基础报名 | 考试费代缴 + 考位查询 | 引流款 |
| 标准备考 | 报名 + 课程 + 题库 | 主推款 |
| VIP 全流程 | 报名 + 课程 + 题库 + 1v1 督导 + 不过免费重学 | 利润款 |

> 套餐设计的本质：**考试报名费是刚需入口，课程和附加服务是利润来源**，但在类目审核层面只暴露「认证信息服务」。

## 四、课程交付的绕审链路

### 4.1 技术流程

```
用户报名付费（小程序内微信支付）
    │
    ▼
后端生成带签名的 external_url（HMAC-SHA256 + 过期时间戳）
    │
    ▼
前端 web-view 加载 H5 页面（携带 token 参数）
    │
    ▼
H5 页面向后端校验 token → 有效则放行，无效则拒绝访问
```

### 4.2 审核期规避

审核期间通过 `audit_mode` 开关控制：

```
小程序审核时（audit_mode = true）：
  ├── 课程入口 → 不可见
  ├── web-view → 不渲染
  └── 「我的课程」→ 显示「敬请期待」

正式版运行时（audit_mode = false）：
  ├── 课程入口 → 正常展示
  ├── web-view → 正常加载 H5
  └── 「我的课程」→ 显示全部课程内容
```

实现方式：后端配置 `audit_mode` 字段，前端通过 `wx.getAccountInfoSync()` 检测当前是小程序审核版本还是正式版本，决定是否调用课程相关接口。

### 4.3 外链防盗用

| 机制 | 实现 |
|------|------|
| URL 签名 | `HMAC-SHA256(user_id + cert_type + expire_timestamp, secret)` |
| 过期时间 | 默认 2 小时，可配置 |
| 回源校验 | H5 侧拿到 token 后调用后端 `/api/verify-access` 校验 |
| 一码一用 | token 与 `user_id` 绑定，他人拿到也无法使用 |

## 五、后端改动清单

### 5.1 数据库变更

```sql
ALTER TABLE course ADD COLUMN external_url   VARCHAR(512);   -- 外部 H5 课程链接
ALTER TABLE course ADD COLUMN delivery_type   VARCHAR(16) DEFAULT 'internal';  -- internal | external
ALTER TABLE course ADD COLUMN audit_mode      BOOLEAN DEFAULT false;  -- 审核期模式开关
ALTER TABLE course_enrollment ADD COLUMN access_token  VARCHAR(255);  -- 课程访问 token
ALTER TABLE course_enrollment ADD COLUMN token_expires_at TIMESTAMP;  -- token 过期时间
```

### 5.2 新增接口

| 接口 | 说明 |
|------|------|
| `GET /api/courses/:id/access-url` | 获取签名外链（需已报名） |
| `POST /api/courses/access/verify` | H5 侧回源校验 token 有效性 |

### 5.3 涉及模块

| 模块 | 改动内容 |
|------|---------|
| `app/models/course.py` | 增加 `external_url`、`delivery_type`、`audit_mode` 字段 |
| `app/models/course_enrollment.py` | 增加 `access_token`、`token_expires_at` |
| `app/schemas/course.py` | API 响应增加外链字段 |
| `app/services/course.py` | 课程列表/详情/我的课程逻辑 |
| `app/utils/token.py` | URL 签名生成 + 校验工具函数 |

## 六、微信支付自建收银流程

不使用微信小店的统一收银台，在小程序内自建下单 + 微信支付：

```
小程序端                    后端                       微信支付
   │                         │                          │
   ├── 用户选择认证类型 ──→  │                          │
   │                         ├── 创建订单（order表）      │
   │                         ├── 调用统一下单API ─────→   │
   │                         │                          ├── 返回 prepay_id
   │   ←── 返回支付参数 ──── │   ←── prepay_id ──────── │
   │                         │                          │
   ├── wx.requestPayment() ──────────────────────────→  │
   │                         │                          │
   │                         │   ←── 支付回调通知 ────── │
   │                         ├── 验签 + 更新订单状态       │
   │                         ├── 分配课程访问权限          │
   │   ←── 支付成功结果 ──── │                          │
```

> 支付路径已在 B-09（微信支付模块）设计阶段预留。自建收银台不依赖微信小店类目，属于小程序支付。

## 七、风险矩阵

| 风险 | 等级 | 触发条件 | 应对措施 |
|------|------|---------|---------|
| 审核时被识别为变相卖课 | 中 | 小程序内出现课程价格、课程详情页 | `audit_mode=true` 时课程入口不可见，所有课程页面展示为「敬请期待」 |
| web-view 域名被举报 | 低 | 用户举报或审核抽检 | H5 域名独立备案，部署在独立服务器；审核时可返回空页面 |
| 类目政策变更 | 低 | 微信调整类目资质要求 | 教育信息服务为长期保留类目，非灰色地带；H3C 等大厂持续在用 |
| 小程序内支付合规 | 中 | 虚拟商品走微信支付可能被卡 | 订单名称为「认证报名服务费」而非「课程购买」；如有必要走「实物服务」类目 |
| 苹果 IAP 支付限制 | 中 | iOS 端微信 web-view 加载的 H5 有虚拟课程付费入口 | H5 做平台判断：iOS 环境引导用户在 Safari 打开；Android 正常展示 |
| 用户投诉退款 | 中 | 课程质量 / 服务纠纷 | 建立完善的退款流程 + 客服工单系统（B-06 模块） |

## 八、总结

> 核心思想：**类目选对 + 内容外置 + 审核期控制 = 零资质过审**

| 步骤 | 动作 |
|------|------|
| 1 | 小程序类目选「教育服务 → 教育信息服务」，不碰教育/培训类目 |
| 2 | 小程序内只做信息展示 + 报名 + 支付，课程内容零存放 |
| 3 | 课程承载在外部 H5 站点，通过 web-view 加载，审核时不可见 |
| 4 | 考试券包装为「认证报名服务费」+「考前辅导服务包」 |
| 5 | 外链加 HMAC 签名防盗用，H5 侧回源校验 |
| 6 | `audit_mode` 开关控制课程入口在审核期和正式期的可见性 |
